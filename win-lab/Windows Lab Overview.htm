Lab Overview

In this lab scenario, you are a consultant assigned to FinanceTech, a FinTech company. FinanceTech develop software for deployment on premise and has started to offer its platform as a SaaS offering initially deployed from its own data-centers but is moving to 3rd party cloud providers such as AWS.

Management recognizes the potential benefits of Ansible and has hired you to develop playbooks/roles to manage Windows Active Directory for Authentication.

Provision Lab Environment

  1. Navigate to the OPENTLC lab portal. (If you don’t know your credentials, visit the OPENTLC Account Management page page)

  2. Log in to the portal using your OPENTLC credentials

  3. Select Services and then Catalogs

  4. Expand the OPENTLC Automation catalog

  5. Order the Ansible Tower Lab to provision your environment.

    1. Select the lab item

    2. Select the "Lab Parameters" tab and choose your Region

    3. Accept terms and conditions if required

    4. Click Submit

  6. Emails will notify you of your GUID (Unique identifier) and environment details.

    1. Further details will be available in the lab instructions.

Deployments can take between 15-45 minutes depending on the software deployed.

Provisioned Environment Hosts

The lab environment consists of 3 servers for Tower Installation, 2 Servers for Postgressql database, 1 Server for Windows Active Directory and single Administration server also refered to as a Bastion/Jumpbox server

  • Administration server: bastion.${GUID}.example.opentlc.com, bastion.${GUID}.internal

  • Ansible Tower Nodes: tower{1,2,3}.${GUID}.example.opentlc.com, tower{1,2,3}.${GUID}.internal

    • There is also an Elastic Load Balancer for the tower servers named, towerlb.${GUID}.example.opentlc.com, to all the tower instances.

  • External Master Postgresql Database: support1.${GUID}.example.opentlc.com, support1.${GUID}.internal

  • External Slave Postgresql Database: support2.${GUID}.example.opentlc.com, support2.${GUID}.internal

  • Windows Server: ad1.${GUID}.internal, ad1.${GUID}.example.opentlc.com

Lab Goals

  • Ping and execute ad hoc commands on each Ansible node and Windows Server Node

  • Create Roles to install Windows Active Directory

  • Create Roles to setup OpenSSH service on Windows Server 2012 host

  • Create roles to create Windows AD User

  • Verify Users are able to login on Windows Server 2012 node with their password.

  • Optimal Module choice for tasks

  • Provide meaningful, and consistent, feedback during playbook execution

Lab: Configure Administration host for Ansible-Windows Management

In this section, you will configure bastion/laptop as control host to manage Windows Server 2012 node.

The FinanceTech company prefers to use Kerberos for authentication.

Windows Server provisioning steps [Review]

WinRM is already configured on Windows Server 2012 node, we ran the following PowerShell script to achieve that: As part of the deployment of this environment, the provisioning included, running the following commands on the windows servers. In the field you would need to run these commands or have them run by the deployment process.

"<powershell>\n",
"$admin = [adsi]('WinNT://./administrator, user')\n",
"$admin.PSBase.Invoke('SetPassword', '{{ windows_password | default(generated_windows_password) }}')\n",
"$scriptPath=((New-Object System.Net.Webclient).DownloadString('https://raw.githubusercontent.com/ansible/ansible/devel/examples/scripts/ConfigureRemotingForAnsible.ps1'))\n",
"Invoke-Command -ScriptBlock ([scriptblock]::Create($scriptPath)) -ArgumentList '-skipNetworkProfileCheck'\n",
"</powershell>"

Prepare Administration host

  1. Install python and Kerberos dependencies to manage a windows hosts from the bastion instance. (python-devel, krb5-devel, krb5-libs, krb5-workstation, python-pip, gcc)

  2. Using pip Install pywinrm newer than version 0.2.2, Python client for the Windows Remote Management. It allows you to invoke commands on a Windows host.

  3. Verify the connectivity to Windows Host.

[root@bastion ~]# ansible windows -m win_ping
ad1.GUID.internal | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
Don’t be concerned with InsecureRequestWarning warnings.

Lab: Install Windows Active Directory Using Ansible Playbooks

In this section, you will install Windows AD.

Windows AD Details:

ParametersValues

AD Admin Password

Get from /etc/ansible/hosts

Domain Name

ad1.${guid}.example.opentlc.com

AD Safe Password

Get from /etc/ansible/hosts

AD Admin User

admin@ad1.${guid}.example.opentlc.com

Create a role to install Active Directory on Windows Server 2012.

  1. In your Ansible project repository, create the role under this path: roles/win_ad_install/tasks/main.yml

    1. Role should install the AD-Domain-Services Windows feature (hint, use the win_feature module)

    2. Role should setup active directory controller based on the information provided above (hint, use the win_domain module)

    3. Role should reboot the instance after configuration is completed (hint, use the win_reboot module)

    4. Role should print out the domain list availble after reboot is completed. (hint, use the shell command: nltest /dclist:{{ ad_domain_name }}")

  2. Create variables file in your project directory called my_ad_vars.yml

    1. The vars file should use the following values:

      1. ad_domain_name: ad1.${guid}.example.opentlc.com

  3. Create default variables file in roles/win_ad_install/vars/main.yml

    1. The vars file should use the following values:

      1. ad_safe_mode_password: "{{ansible_password}}"

      2. ad_admin_user: "admin@{{ ad_domain_name}}"

      3. ad_admin_password: "{{ansible_password}}"

  4. Create a playbook to apply your role to the windows host group

    1. Name that playbook: ad.yml

  5. Run the playbook to install and configure Active Directory on your windows hosts:

    [root@bastion ~]# ansible-playbook ad.yml
Running this playbook might take a while

Lab: Configure Control host for Kerberos Authentication [ Guided Exercise ]

In this, section you will configure laptop/bastion host for Kerberos Authentication.

  1. Install python kerberos, for authentication.

    [root@bastion ~]# pip install pywinrm[kerberos]

  2. Create a file and specify Windows AD domains you are going to authenticate against

    [root@bastion ~]# export GUID=`hostname | awk -F"." '{print $2}'`
[root@bastion ~]# export GUID_CAP=`echo ${GUID} | tr 'a-z' 'A-Z'`
[root@bastion ~]# cat << EOF > /etc/krb5.conf.d/ansible.conf

[realms]

 AD1.${GUID_CAP}.EXAMPLE.OPENTLC.COM = {

 kdc = ad1.${GUID}.example.opentlc.com
 }

[domain_realm]
 .ad1.${GUID}.example.opentlc.com = AD1.${GUID_CAP}.EXAMPLE.OPENTLC.COM
EOF

  3. Verify that you are able to authenticate as Administrator, the password is available in the /etc/ansible/hosts file

    [root@bastion ~]# kinit administrator@AD1.${GUID_CAP}.EXAMPLE.OPENTLC.COM

Lab: Create a Role to Configure OpenSSH service on Windows Server 2012.

Create a role to install packages on a Windows Server

In this section, you will write a playbook to configure OpenSSH services on Windows Server host.

It’s highly recommended to read the documentation, misconfiguration might require your to RDP into the instance to solve the issue prior to OPENSSH installation. Specifically, the "/SSHServerFeature" parameter is very important to remember.

  1. In your Ansible project repository, create the role under this path: roles/win_package_install/tasks/main.yml

    1. Role should be able to accept a list/array of packages and install them (hint, use the win_chocolatey module and with_items)

    2. ansible_distribution should be defined as Microsoft Windows Server 2012 R2 Standard

Create a role to enable services on a Windows Server

  1. In your Ansible project repository, create the role under this path: roles/win_service_manage/tasks/main.yml

    1. Role should be able to accept a service_name variable and enable it

    2. ansible_distribution should be defined as Microsoft Windows Server 2012 R2 Standard

Create a role to add allow ingress traffic through Windows firewall

  1. In your Ansible project repository, create the role under this path: roles/win_firewall/tasks/main.yml

    1. Role should be able to accept a service_name create a firewall rule based on that name

    2. Allow ingress traffic to port based on the variable local port and the protocol based on the variable protocol_name

Create a playbook to install OPENSSH using the roles created

  1. Create vars file to set values of the variables.

    [root@bastion ~]# cat << EOF > ssh_var.yml
package_name: openssh
parameters: /SSHServerFeature
service_name: SSHD
local_port: 22
protocol_name: tcp
EOF

  2. Create a playbook called win_ssh_server.yml to deploy OPENSSH on the windows hosts group

    1. Playbook should run on the windows hosts group

    2. vars_files should be defined to use the ./ssh_var.yml

    3. Playbook should run the roles: win_package_install, win_service_manage, win_firewall

  3. Test your playbook using the following command:

[root@bastion ~]# ansible-playbook win_ssh_server.yml

  1. Use pip to upgrade Ansible to version 2.4.0

  2. Create a vars file that represents the users in the table above.

    1. Vars file should have a list variable called user_info

  3. In your Ansible project repository, create the role under this path: roles/win_ad_user/tasks/main.yml
